# BetterLB Architecture Documentation

> **Note:** Generated by Maestro | Last updated: 2026-03-03T12:00:00.000Z

## Overview

**BetterLB** is a municipal government portal for Los Baños, Laguna, Philippines - a fork and localization of BetterGov.ph. It provides transparent access to government services, legislative documents, elected officials, departments, and municipal data.

**Architecture Type:** JAMstack (Static Frontend + Serverless Backend)
**Deployment:** Cloudflare Pages (Frontend) + Cloudflare Pages Functions (API)
**Database:** Cloudflare D1 (SQLite-based, edge-native)
**Search:** Meilisearch (self-hosted)
**Languages:** TypeScript (Frontend), Python (Data Pipeline), SQL (Database)

## Frontend Architecture (React/Vite)

### Technology Stack
- **Runtime:** React 19 with TypeScript strict mode
- **Build Tool:** Vite 6
- **Styling:** Tailwind CSS v4 with Kapwa Design System fork (@betterlb/kapwa)
- **Routing:** React Router DOM v6 (file-based routing)
- **State Management:** React hooks (useState, useContext) + URL params (nuqs)
- **Internationalization:** i18next (English/Filipino, namespace-based)
- **Maps:** Leaflet + React Leaflet
- **Charts:** Recharts
- **Forms:** React Hook Form + Zod validation
- **Testing:** Vitest (unit), Playwright (E2E)

**Tailwind v4 Prefix Rules for Semantic Tokens:**
- Text: `text-kapwa-text-*` (e.g., `text-kapwa-text-strong`)
- Background: `bg-kapwa-bg-*` (e.g., `bg-kapwa-bg-surface`)
- Border: `border-kapwa-border-*` (e.g., `border-kapwa-border-weak`)
- Typography/Spacing: No prefix (e.g., `kapwa-heading-md`, `p-kapwa-lg`)

### Navigation Page Architecture

BetterLB uses a unified navigation page design system across all content sections (services, government, statistics, transparency, OpenLGU).

**Scope:**
- Navigation pages use sidebar navigation with hierarchical content organization
- Includes: `/services/*`, `/government/*`, `/statistics/*`, `/transparency/*`, `/openlgu/*`
- Excludes: Homepage (`/`), Admin dashboard (`/admin/*`)

**Design Token Standards:**

All navigation pages MUST use Kapwa semantic tokens following Tailwind v4 prefix conventions:

| Context | Background Class | Text Color | Usage |
|---------|-----------------|------------|-------|
| Page container | `bg-kapwa-bg-surface` | `text-kapwa-text-default` | All navigation pages |
| Hero header (index) | `bg-kapwa-bg-surface-bold` | `text-kapwa-text-strong` | Index page headers |
| Compact header (detail) | `bg-kapwa-bg-surface` | `text-kapwa-text-strong` | Detail page headers |
| Section header | `bg-kapwa-bg-hover-weak` | `text-kapwa-text-strong` | Sub-sections within pages |
| Sidebar container | `bg-kapwa-bg-surface` | `text-kapwa-text-default` | Sidebar background |
| Active nav item | `bg-kapwa-bg-selected` | `text-kapwa-text-strong` | Currently selected item |
| Hover nav item | `hover:bg-kapwa-bg-hover` | `text-kapwa-text-default` | Interactive state |

**Key Rules:**
- ALL navigation pages use `bg-kapwa-bg-surface` (NO raw color tokens like `bg-gray-900`)
- NO arbitrary spacing values (use Kapwa spacing tokens: `p-kapwa-lg`, `gap-kapwa-md`)
- ALL text uses Kapwa typography tokens (`kapwa-heading-*`, `kapwa-body-*`)

**Layout Patterns:**

**Pattern A: Index Page with Sidebar**
```tsx
<SidebarLayout
  collapsible={true}
  defaultCollapsed={false}
  headerNode={
    <PageHeader
      variant="hero"
      title="Page Title"
      autoBreadcrumbs={true}
    />
  }
  sidebar={<SectionSidebar />}
>
  <Outlet />
</SidebarLayout>
```

**Pattern B: Detail Page with Sidebar**
```tsx
<SidebarLayout
  collapsible={true}
  defaultCollapsed={true}
  headerNode={
    <PageHeader
      variant="compact"
      title="Item Title"
      autoBreadcrumbs={true}
    />
  }
  sidebar={<SectionSidebar />}
>
  <Outlet />
</SidebarLayout>
```

**Component Specifications:**

**PageHeader Component:**
- `variant`: "hero" (index pages) or "compact" (detail pages)
- `autoBreadcrumbs`: ALWAYS use `true` (NEVER manual breadcrumb components)
- `centered`: Optional for hub pages (e.g., `/government`)

**SidebarLayout Component:**
- `collapsible`: `true` for index/detail pages, `false` for hub pages
- `defaultCollapsed`: `false` for index pages, `true` for detail pages
- Backgrounds: `bg-kapwa-bg-surface`
- Borders: `border-kapwa-border-weak`

**Responsive Behavior:**
- Mobile (< 640px): Sidebar hidden by default, hamburger menu toggle (44px minimum touch target)
- Tablet (640px - 1024px): Collapsible sidebar, overlay when expanded
- Desktop (> 1024px): Sidebar visible by default (index pages)

**Accessibility Requirements:**
- WCAG 2.1 Level AA compliance
- Keyboard navigation (Tab, Enter, Escape, Arrow keys)
- ARIA attributes: `aria-label`, `aria-current="page"`, `aria-expanded`
- Focus indicators: `focus-visible:ring-2 focus-visible:ring-kapwa-border-focus`
- Semantic HTML: Proper heading hierarchy (h1 → h2 → h3), no skipped levels

**Utility Functions:**

**Best Practices:**
1. **Always use `ChartTooltip`** - Provides consistent styling, sorting, and accessibility
2. **Use `standardAxisProps`** - Ensures consistent axis typography across all charts
3. **Import from `@/constants/charts`** - Centralized theme prevents inconsistencies
4. **Use `ResponsiveChart` inside DetailSection** - Avoids double card styling
5. **Use `ChartContainer` for standalone cards** - Provides proper borders and shadows
6. **Define TypeScript interfaces** - Never use `any` for chart data structures
7. **Use StatCard for KPIs** - Consistent KPI display with trend indicators
8. **Format values with utilities** - Use `formatPesoAdaptive()` for currency
9. **Test with real data** - Ensure charts render correctly with edge cases (null values, empty arrays)
10. **Color contrast matters** - Test colors for WCAG AA compliance
11. **Responsive design** - Test on mobile (375px) and desktop (1920px+)
12. **ARIA labels required** - All ChartContainer instances need descriptive titles

## Data Pipeline Architecture

### Pipeline Overview

The data pipeline transforms raw legislative PDFs into structured JSON data.

**Technology:**
- Python 3.x
- PDF extraction: `pdfplumber`, `PyPDF2`, `pdf2image`
- Validation: `jsonschema`, `aiohttp`

### Pipeline Stages

```
┌─────────────────┐    ┌─────────────────┐    ┌─────────────────┐
│   1_scrape.py   │───▶│ 1.5_normalize   │───▶│   2_download.py  │
│  (Web Scraping) │    │   (Normalization)│   │  (PDF Download) │
└─────────────────┘    └─────────────────┘    └─────────────────┘
                                                        │
                                                        ▼
┌─────────────────┐    ┌─────────────────┐    ┌─────────────────┐
│   4_generate.py  │◀───│   3_parse.py    │◀───│  PDF Files       │
│ (JSON Output)   │    │  (PDF Parsing)  │    │  (Local Storage) │
└─────────────────┘    └─────────────────┘    └─────────────────┘
```

### Stage Details

**1. `1_scrape.py` - Web Scraping**
- Scrapes official Los Baños legislative portal
- Discovers available documents
- Extracts metadata (numbers, dates, titles)

**1.5. `1.5_normalize.py` - Data Normalization**
- Standardizes date formats (ISO 8601)
- Normalizes document numbering
- Removes extra whitespace

**2. `2_download.py` - File Download**
- Downloads PDFs to `pipeline/data/pdfs/`
- Retries failed downloads
- Skips already downloaded files

**3. `3_parse.py` - PDF Parsing**
- Extracts text from PDFs
- Handles OCR for scanned documents
- Identifies document metadata

**4. `4_generate.py` - JSON Generation**
- Converts parsed data to JSON schema
- Generates structured output
- Outputs to `src/data/legislation/`

## Architecture Decision Records

### ADR-001: JAMstack over Monolith
**Decision:** Use JAMstack (Cloudflare Pages + Functions) instead of traditional monolithic server.

**Rationale:**
- Edge deployment (low latency globally)
- Serverless scaling (pay-per-use)
- Simplified operations (no server management)
- Built-in DDoS protection

**Trade-offs:**
- Limited runtime (Edge vs Node.js full API)
- Cold starts on first request
- Database connection limits (D1)

### ADR-002: D1 over PostgreSQL
**Decision:** Use Cloudflare D1 (SQLite) instead of PostgreSQL.

**Rationale:**
- Edge-native (replicated globally)
- Single database for local and remote
- Seamless integration with Pages Functions
- Lower cost at low traffic

**Trade-offs:**
- Less feature-rich than PostgreSQL
- Limited connection pooling
- Single-region writes

### ADR-003: Kapwa Fork over Upstream
**Decision:** Fork Kapwa Design System instead of using upstream bettergovph/kapwa.

**Rationale:**
- Municipal branding requirements
- Custom color scheme
- Local component needs

**Trade-offs:**
- Maintenance burden (merge upstream changes)
- Divergence over time

### ADR-004: Absent-Only Attendance
**Decision:** Record only absences, not presence, for legislative sessions.

**Rationale:**
- Simpler data entry (fewer records)
- Assumes present unless marked absent
- Reduces manual entry workload

**Trade-offs:**
- Cannot distinguish "present but not voting"
- Requires total member count for calculation

### ADR-005: Static JSON over CMS
**Decision:** Store directory data (departments, barangays) as JSON files instead of CMS.

**Rationale:**
- Version control (Git history)
- No database queries for static data
- Faster page loads
- Easier content reviews

**Trade-offs:**
- Requires rebuild for content changes
- No non-technical editor access
- Larger bundle size
