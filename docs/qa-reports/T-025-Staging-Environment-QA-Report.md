## Current Deployment Architecture

|-------------|---------|-----|---------|

### Deployment Workflow

**File:** `.github/workflows/deploy.yml`

```yaml
deploy:
  environment: ${{ github.event_name == 'pull_request' && 'preview' || 'production' }}
  needs: quality-gate
  steps:
    - name: Deploy to Cloudflare Pages
      uses: cloudflare/wrangler-action@v3.9.0
      with:
        command: pages deploy dist --project-name=betterlb
```

**Current Behavior:**
- PRs deploy to temporary preview URLs
- Merged PRs deploy directly to production
- No intermediate staging environment

## What's Missing: Staging Environment

A proper staging environment would include:

### 1. **Staging Deployment Workflow** ❌ Missing
- Should deploy from a `staging` or `develop` branch
- Separate workflow file or job in existing deploy.yml
- Should run after quality checks but never auto-deploy to production

### 2. **Staging Cloudflare Pages Project** ❌ Missing
```bash
# Should exist but doesn't:
npx wrangler pages project create betterlb-staging --production-branch=staging
```

Current projects:
```
betterlb     → betterlb.pages.dev (production)
betteruplb   → betteruplb.pages.dev (unrelated)
lhs-hoa      → lhs-hoa.pages.dev (unrelated)
```

### 3. **Staging D1 Database** ❌ Missing
Should have:
```json
{
  "binding": "BETTERLB_DB",
  "database_name": "betterlb_openlgu_staging",
  "database_id": "<new-staging-db-id>",
  "migrations_dir": "db/migrations"
}
```

Current setup only has:
- `betterlb_openlgu` (production, 4.33 MB, 17 tables)

### 4. **Staging KV Namespace** ❌ Missing
Should have:
```json
{
  "binding": "WEATHER_KV",
  "id": "<new-staging-kv-id>"
}
```

Current setup only has production KV.

### 5. **Staging Environment Variables** ❌ Missing
Staging would need its own set of secrets:
- `GITHUB_CLIENT_ID_STAGING`
- `GITHUB_CLIENT_SECRET_STAGING`
- `GOOGLE_CLIENT_ID_STAGING`
- `GOOGLE_CLIENT_SECRET_STAGING`
- etc.

### 6. **Branch Strategy** ❌ Not Documented
No documentation on:
- Which branch triggers staging deployment
- Promotion workflow from staging → production
- Branch protection rules for staging

## Expected Staging Configuration

### Recommended Deployment Workflow

```yaml
# .github/workflows/deploy-staging.yml (new file)
name: Deploy to Staging
on:
  push:
    branches: [staging]

jobs:
  deploy-staging:
    runs-on: ubuntu-latest
    environment: staging
    needs: quality-gate
    steps:
      - name: Deploy to Cloudflare Pages (Staging)
        uses: cloudflare/wrangler-action@v3.9.0
        with:
          command: pages deploy dist --project-name=betterlb-staging
```

### Recommended Wrangler Configuration

```json
// wrangler.jsonc (staging environment)
{
  "name": "betterlb-staging",
  "pages_build_output_dir": "dist",
  "compatibility_date": "2026-01-13",
  "kv_namespaces": [
    {
      "binding": "WEATHER_KV",
      "id": "<staging-kv-id>"
    }
  ],
  "d1_databases": [
    {
      "binding": "BETTERLB_DB",
      "database_name": "betterlb_openlgu_staging",
      "database_id": "<staging-db-id>",
      "migrations_dir": "db/migrations",
      "remote": true
    }
  ]
}
```

### Recommended Branch Strategy

```
main → Production (betterlb.pages.dev)
  ↑
staging → Staging (betterlb-staging.pages.dev)
  ↑
feature/* → Preview (PR deployments)
```

### Required Actions

**Option 1: Define Clear Staging Requirements (Recommended)**
Before implementing, clarify:
1. **Branch strategy:** Is there a `staging` branch? Or `develop`?
2. **Promotion workflow:** How does code move from staging → production? (Manual approval? Automatic merge?)
3. **Environment isolation:** Does staging need separate databases/KV, or can it share with production?
4. **URL:** What should the staging URL be? (betterlb-staging.pages.dev?)
5. **Data seeding:** Should staging have test data? Copies of production data? Empty?
6. **Access control:** Who has access to staging? Public or authenticated only?

**Option 2: Close Task as Won't Fix**

- Preview environments provide pre-merge testing
- Quality gates prevent broken deployments
- Small team may not need persistent staging
- Preview URLs serve the same purpose as staging

**Option 3: Implement Minimal Staging**
If staging is needed:
1. Create `betterlb-staging` Cloudflare Pages project
2. Add deploy-staging.yml workflow for `staging` branch
3. Create staging D1 database
4. Document staging workflow in CI-CD-SETUP.md
